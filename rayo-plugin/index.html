Rayo plugin

Product

Rayo plugin enables designers to pull a library of assets and metadata from the Bauer API, and applies directly onto designs.

Role

Research, Product Design, Testing, Development

Overview

As someone who is always looking for ways to streamline or automate processes, I initiated this project with an aim to help my team speed up the UI design tasks, such as explorations, creating user testing mockups, building production ready designs.

Problem

It requires too much effort from the design team to build and maintain a large library of assets in Figma for many different brands from multiple regions that change overtime.

Goal

To build a Figma plugin that can provide live assets from all of the brands from all of the regions we serve.

Why

Besides speeding up the making of our production designs, there is another major reason why we want to build our designs as accurate as possible.

We test our designs with some of our loyal listeners who are very passionate about their favourite stations. Despite our disclaimer about the prototype was only put together for testing specific features, not everything works and might not reflect the actual application. They always notice abnormality and would point out things such as, “this show is not suppose to be here”. We now build accurate designs to avoid these kinds of feedback that could waste valuable testing time.

POC

I began with running a local server pulling images from a folder with thumbnail images. The plugin doesn’t even have UI. When initialed, it updates all the layers that matched the required layer names within the selected frame(s).

Navigation

I took a lot of inspiration from Music Data. It is a plugin that allows users to search for songs and update designs with album artworks, title, artists etc.

Since not everyone is familiar with all of the station names, I instead build a dropdown to list out stations for my team to choose from.

Setting up the logic

I defined requirements for the plugin to identify the correct layers to update. E.g. A shape layer with the name show-thumbnail, the fill will be replaced with the selected radio show thumbnail image. A text layer with the name show-name, the text string will be replaced with the title of the radio show. I added more as I build more functions into the plugin, but the naming principles are the same.

I built a notification prompt to appear when none of the selected layers matched the requirements, it lists out the layer names required.

I also wrote a documentation in Confluence to explain how the codes were structured, and instructions on layer names and how it works.

Connecting to the API

With my limited coding skill, I reached out to a full stack developer I worked closely for assistance. He helped me with the CORS issues, and provided the appropriate API to be used to fetch stations and shows.

Adding podcasts

My teammates were working on the Podcast V2 designs, and was thinking it would be nice to have podcasts available in the plugin.

When I was adding podcasts, knowing that the Rayo app can filter podcasts by genres, meaning we have to mockup designs that display podcasts from a specific genre. To make our lives easier, I also build a genre filter in the plugin.

Other regions

Rayo is going be available for 8 regions outside the UK: Ireland, Poland, Portugal, Slovakia, Denmark, Sweden, Finland and Norway. Each have slightly different configurations based on the UK Rayo app.

I was tasked with designing the For You page (personalised homepage). To make my life easier, I want the plugin to be able to provide assets from other regions as well. I got the API for other regions by reaching out to the iOS dev who was building the POC of the Rayo app for other regions.

Due to the fact that APIs from some of the regions are not ready, I decided to hard coded the available ones in, and add more when the others are ready.

Episode data

Since the beginning of the project, I always wanted to make episode level data available because there are quite a few components that display episode metadata. I just didn’t know how to build it in. However, as I got more familiar with the structure of the APIs, I was more confident to build logic to fetch assets and data from multiple APIs to make it work.

Always evolving

I am always thinking how to help the team and to make our work easier. The next step is to find a way to populate the whole page with metadata based on requirements with a click of a button. Stay tune for more.
